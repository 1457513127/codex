# 小范围变更政策

## 目标

让每次迭代可理解、可验证、可回退，避免项目讨论增多后出现范围漂移或界面风格分裂。

## 每轮工作协议

1. 读取 `AGENTS.md`、项目持续记忆和最近更新日志。
2. 将本轮需求压缩为一个可验收目标，并在修改前说明涉及文件。
3. 只修改实现该目标所需的最少文件，不顺带清理无关代码。
4. 涉及界面时先读取 `design-system/README.md`，只使用共享 tokens。
5. 完成后执行适合该范围的检查，更新本地日志，并同步 Notion。

## 默认变更预算

| 类型 | 默认上限 |
| --- | ---: |
| 既有业务或界面文件 | 5 个 |
| 既有代码修改行数 | 200 行 |
| 每次更新的用户目标 | 1 个 |
| 新增视觉 token | 仅在现有 token 无法表达需求时增加，并记录理由 |

超出上限不是禁止，但必须在编辑前向用户说明原因和拟扩大的文件范围。

## 提交约定

- 提交信息说明单一结果，例如 `接入 Airbnb 设计系统基线`。
- 不把尚未确认的产品决策与已确认开发改动放进同一提交。
- 每次提交在 `docs/UPDATE_LOG.md` 和 Notion 文档中保留同一条更新摘要。
