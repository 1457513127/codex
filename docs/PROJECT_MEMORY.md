## 稳定结论

| 项目 | 当前约束 |
| --- | --- |
| 项目事实源 | 以仓库文件为准，不依赖单一 Codex 对话记忆 |
| 产品定位 | 和趣是面向私域社群经营者的社群变现平台，服务已有社群经营者和零基础新手群主，把微信群、兴趣圈层和本地社区里的信任关系转化为可发布、可成交、可履约、可复购的社群生意 |
| 核心价值表达 | 和趣不是单一商品卖货工具或单纯分销平台，而是以“社群经营者 + 可信供给 + 消费者订单 + 平台运营监督”组织交易，把群聊里的口头信任交易变成有规则、有凭证、有售后和有收益解释的交易闭环 |
| 当前主线 | 选择经营方向 -> 社群冷启动 -> 建店 -> 商品库/供给库 -> 开团发布 -> 分享成交 -> 履约售后 -> 收益结算 -> 复购经营 -> 选品中心放大 |
| 产品承载 | 项目标题和开场主题不突出双通道；V1 由同一微信小程序承载 C 端消费通道 + B 端团长/群主经营通道 |
| 多端口径 | 后续建设团长 App 作为 B 端专业经营工具，并扩展微信/小红书/抖音小程序消费端；多端共用交易底座 |
| 新手群主口径 | 新手群主是核心客户之一；首发“起号”定义为社群冷启动，不做公域账号代运营 |
| 用户分层口径 | 已有社群经营者侧重从 1 到 N，重点是发布效率、成交追踪、履约售后和收益解释；零基础新手群主侧重从 0 到 1，重点是服务谁、如何承接社群和如何完成第一笔真实交易 |
| 角色边界 | 供给方负责供给真实性、交付能力和基础履约承诺；社群经营者负责社群信任经营和交易触达；消费者需要购买须知、履约凭证和售后入口；平台运营负责准入审核、风险暂停、纠纷协同和账本监督 |
| 选品中心口径 | 选品中心是 V2 增强版能力；V1 只保留商品库/供给库、审核货盘和人工推荐 |
| 发布模式 | 正式采用“商品库/供给库 + 开团类型”，不再以割裂的创建活动/创建商品长表单作为主入口 |
| 变现场景优先级 | P0 首发主验证为活动变现和卡券变现；P1 为低风险电商对照验证；本地服务为受控后置扩展；内容/课程暂不作为 V1 独立变现场景 |
| 业务主线口径 | 变现主线按“经营方向 -> 社群冷启动/建店 -> 审核供给 -> 开团发布 -> 分享成交 -> 履约售后 -> 收益结算 -> 复购经营”串联，先跑通低风险交易闭环，再扩展供给丰富度 |
| 开团类型 | 普通团购、预售团购、卡券团购、活动报名、到店服务 |
| 商品库分类 | 活动票务、实物商品、自提商品、虚拟卡券、到店服务 |
| 活动货盘口径 | 首发白名单覆盖亲子家庭、青年兴趣、创业职场、社区生活；受控扩展文旅休闲、健康运动、学习成长；暂不开放高风险户外、多日复杂行程、强效果承诺、医疗健康、金融投资和未成年人高风险托管活动 |
| 电商变现口径 | V1 只做低风险标准商品和自提商品，作为社群交易对照验证；食品生鲜、强售后品类和复杂供应链不进入首发主线 |
| 卡券与本地服务口径 | 卡券是首发标准化交易场景，用于验证凭证、有效期、核销、售后和复购；本地服务先限于到店预约/核销，规模化服务供给、上门服务和复杂履约后置 |
| 内容/课程口径 | 内容/课程不作为 V1 独立场景；体验课、工作坊可作为活动或卡券供给进入白名单，纯线上内容付费、系统课、讲师分账和强效果承诺培训后置 |
| 订单类型 | 活动报名订单、物流订单、自提订单、卡券核销订单、服务预约订单 |
| 分销边界 | 首发只做一级固定佣金分享赚和一级加价帮卖 |
| 当前护城河口径 | 通过售后服务台和收益账本，把履约、退款、责任协同和结算监督纳入平台闭环 |
| 市场口径 | 使用 TAM/SAM/SOM；市场数据只证明方向值得验证，不证明收入或份额 |
| 竞品参考口径 | 快团团、群接龙、有赞和本地生活平台只作为产品组织方式参考；和趣可借鉴供给库、开团、帮卖、订单、卡券核销和团长业绩统计，不照搬全品类开放、多级裂变、成熟商家重后台、公域达人投放或未经核验的即时结算承诺 |

## 协作结论

- 采用 `一个项目仓库 + 多个专题对话 + 文件化长期记忆`。
- 新专题对话结束时，必须把稳定结论、决策或待确认问题写回项目文件。
- PPT、原型、主讲材料属于交付物，不在纯讨论阶段自动修改。
- 每次有效更新后，更新本地日志、同步 Notion、创建聚焦提交并推送 GitHub。
