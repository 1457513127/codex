# 和趣项目会演示界面走查脚本

版本：V1.0  
日期：2026-05-26  
演示入口：[和趣项目会演示原型.html](./和趣项目会演示原型.html)

## 1. 演示目的

本原型不是研发验收页面，而是帮助运营和开发在项目会上判断：

- 家长是否能在团长分享场景中快速知道“谁推荐、谁履约、在哪里售后”。
- 团长是否真正需要一个轻主页和可查佣金，而不是完整社群社区。
- 退款风险与佣金延迟结算是否被产品路径正确表达。
- 运营能否对允许分销的活动和高退款供给进行控制。

## 2. 建议演示顺序

### 场景 1：家长从微信群进入团长主页

操作：点击导航中的 `家长看到的团长主页`。

重点观察：

- 首屏出现团长名称和审核状态。
- 明确写出团长是推荐者，主办方提供服务，订单售后由和趣承接。
- 家长可以在一个货架看到多个当前可报名活动，而非逐条翻群链接。

会议追问：

- 运营：家长现实中最想在主页看见哪些信息？现有群里最常问什么？
- 开发：主页分享参数和访问埋点能否支持团长唯一归因？

### 场景 2：家长查看活动详情

操作：在主页点击 `查看详情` 或导航中的 `活动详情与支付前披露`。

重点观察：

- `实际主办方` 和 `推荐团长` 分开展示。
- 适龄、陪同、安全、退款、保险说明在支付前可见。
- 支付按钮上方再次提示订单和售后留在平台。

会议追问：

- 运营：外部精选活动是否都能提供这些字段？
- 开发：原有报名订单是否能保存披露与规则版本快照？

### 场景 3：团长管理推广活动

操作：点击 `团长推广工作台`，再进入 `选择可推广活动`。

重点观察：

- 团长入口不再出现发布商品、优惠券或邀请下级。
- 团长只可选择运营审核通过且允许推广的活动。
- 活动卡片显示主办方、退款风险提示和预估佣金规则。

会议追问：

- 运营：哪些活动不允许帮卖？由谁关闭分销？
- 开发：团长上架/下架货架和分享入口如何实现？

### 场景 4：佣金状态与退款

操作：从团长工作台进入 `佣金明细与退款影响`。

重点观察：

- 佣金状态明确分为预估、待生效、可结算、已结算和已回退。
- 页面解释：活动完成并超过退款窗口后才可结算。
- 退款订单会取消佣金或进入回退/冲抵处理中。

会议追问：

- 支付/法务：团长结算主体是什么？已结算退款路径是否成立？
- 运营：履约后延迟结算是否可以向团长解释并接受？

### 场景 5：运营控制供给与分销质量

操作：点击 `运营审核与质量看板`。

重点观察：

- 看板不是只展示成交金额，同时展示退款订单率、归因完整率和佣金待处理。
- 运营可对退款风险高的活动执行 `暂停分销`。
- 审核区明确首轮只开放亲子活动，商品与泛服务保持未开放。

会议追问：

- 运营：首轮 20 场活动如何选择？高退款活动是否存在必须保留的理由？
- 开发：数据看板和导出最小做到什么程度可支持真实试点？

## 3. 页面与开发范围对应表

| 演示页面 | 首期是否开发 | 核心数据/接口 |
| --- | ---: | --- |
| 团长主页（家长端） | 是，P0 | 团长资料、活动货架、分享归因 |
| 活动详情披露 | 是，P0 | 主办方、团长来源、规则快照、保险说明 |
| 团长推广工作台 | 是，P0 | 主页状态、订单、佣金摘要 |
| 可推广活动 | 是，P0 | 允许分销、团长上架、佣金规则 |
| 佣金明细 | 是，P0 | 佣金状态机、结算批次、退款影响 |
| 运营质量看板 | 基础版，P0/P1 | 活动/团长审核、归因导出、退款与暂停分销 |
| 内容社区/商品选品 | 否 | 不进入当前试点 |

## 4. 会议现场记录重点

| 页面 | 记录什么反馈 |
| --- | --- |
| 主页 | 家长必要信息、团长身份展示、是否需要筛选 |
| 详情 | 主办方资料可获得性、退改和保险文案 |
| 团长工作台 | 团长最常动作、佣金接受方式 |
| 运营看板 | 白名单/暂停规则、报表字段、人工处理成本 |
| 结算 | 技术/支付不可行项与替代方案 |

## 5. 使用提醒

- 原型中的金额、活动和人员均为演示数据，不代表真实经营结果。
- 原型刻意没有商品、优惠券、多级分销或内容社区入口，这是产品范围判断，不是遗漏。
- 会议结束后，应将通过或否决的页面范围更新到产品规划和研发估算中。
