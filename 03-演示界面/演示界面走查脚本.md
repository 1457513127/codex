## 1. 演示定位

本原型用于同步首发产品形态：团长按社群类型建店，从审核货盘发布或选卖活动、电商与到店本地服务，消费者完成交易后进入订单服务台，平台围绕履约、售后、退款和收益结算进行监督。

会议中统一强调：

- 首发重点不是大群社区、内容运营或泛分销，而是交易履约闭环。
- 三类首发交易入口为 `活动`、`电商`、`本地服务`，均以低风险白名单与审核控制范围。
- 活动是主验证场景，电商是对照验证场景，本地服务是受控补充场景。
- 推广支持 `固定佣金分享赚` 或 `加价帮卖`，两者互斥且只允许一个直接推广团长。
- 商单讨论中的多级转单、经销商层级、保证金和会员可见只进入后续扩展池。

## 2. 主讲路径

### 路径 1：产品总览

操作：选择左侧 `产品总览`。

讲解要点：

- 产品主线升级为“社群建店 -> 发布/选卖 -> 分享成交 -> 订单服务台 -> 履约售后 -> 收益结算”。
- `新录音.pdf` 中的活动小群概念，被收敛成订单服务台和双客服响应机制。
- `商单系统功能及抽成规则讨论.pdf` 只提取一级收益、优惠和退款规则，复杂商单能力不进入首发。

### 路径 2：团长端

操作：选择 `团长端`，或在总览手机页点击 `查看团长经营入口`。

讲解要点：

- 团长经营工作台包含收入、订单/报名、核销、社群设置、邀请帮卖和客服。
- 邀请帮卖是一级推广入口，不是发展下级或多级分销入口。
- 团长端要优先处理今天的订单、核销、售后和收益，不做后台大全。

### 路径 3：团员端

操作：选择 `团员端`。

讲解要点：

- 团员中心按履约类型拆分为活动订单、卡券订单、物流订单、自提订单。
- 用户购买后再次进入，应优先看到订单状态、待使用、退款和客服，而不是重新回到销售页。
- 消费者履约体验会直接影响平台信任。

### 路径 4：创建活动

操作：选择 `创建活动`。

讲解要点：

- 活动仍是主验证场景。
- 发布表单必须包含活动信息、地点、场次、票种/分组、报名截止、退改规则、报名信息、保险和核销方式。
- 购买须知和退款规则要结构化，避免后续售后解释成本失控。

### 路径 5：创建商品

操作：选择 `创建商品`。

讲解要点：

- 电商作为对照验证，首发同时覆盖物流、自提和虚拟权益。
- 商品发布必须明确售卖方式、售后规则、规格、库存、限购和购买需填信息。
- 不首发食品生鲜、美容医疗、责任不清品类。

### 路径 6：详情/须知

操作：选择 `详情/须知`，或在创建活动/创建商品页点击发布。

讲解要点：

- 详情页既承担转化，也承担规则披露。
- 价格、报名截止、履约方、社群来源、咨询入口和立即报名/购买必须清楚。
- 购买须知不是附录，退改比例、报名资料、保险和核销规则直接影响售后。

### 路径 7：订单详情

操作：选择 `订单详情`，或在详情页点击 `立即报名`。

讲解要点：

- 订单详情展示已支付、待入账、订单金额、优惠金额、报名信息和保险信息。
- 待入账代表售后未闭环前，相关收益不应被当成可自由结算。
- 退款入口必须可见，把退款处理纳入平台规则，而不是让用户绕回私聊。

### 路径 8：服务台

操作：选择 `服务台`，或在订单详情点击 `退款 / 售后`。

讲解要点：

- 订单服务台替代大群，围绕一笔订单组织官方客服、发布方客服和消费者处理问题。
- 售后未解决时，平台可以阻断或延后收益结算。
- 全额退款全链取消；部分退款按比例冲回平台费、发布收益、推广收益和优惠分摊。

## 3. 页面验收表

| 页面 | 必须可见的业务表达 | 当前属性 |
| --- | --- | --- |
| 产品总览 | 交易履约闭环、订单服务台、首发/后置边界 | 产品主线 |
| 团长端 | 收入、订单/报名、核销、邀请帮卖、客服 | 经营入口 |
| 团员端 | 活动/卡券/物流/自提订单分类 | 履约入口 |
| 创建活动 | 场次、票种、报名信息、保险、退改、核销 | 活动主验证 |
| 创建商品 | 售卖方式、售后、规格、库存、购买需填信息 | 电商对照验证 |
| 详情/须知 | 履约方、社群来源、购买须知、咨询和 CTA | 转化与披露 |
| 订单详情 | 支付、待入账、金额、报名/保险、退款入口 | 售后依据 |
| 服务台 | 双客服响应、工单、结算阻断、退款回退 | 平台护城河 |

## 4. 展示提醒

- 原型展示的是目标产品与产品路线，不表示真实小程序后端、自动分账或供给审核能力已经实现。
- 页面继续使用 Airbnb inspired 共享 tokens，不新增第二套视觉规范。
- 原型中的供给、价格、收益与优惠为演示数据，不作为经营承诺。
