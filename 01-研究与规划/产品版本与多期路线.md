# 和趣产品版本与多期路线

日期：2026-05-28

状态：规划假设，随真实验证滚动调整

本文只定义 V0/V1/V2/V3/V4 的阶段目标、功能边界、验证指标、进入下一阶段条件和能力前后置关系，不展开页面交互、字段细节或技术实现。当前产品策划仍在调整期，以下内容用于约束版本节奏，不把未确认功能写成承诺。

## 1. 版本总原则

- 版本不是按日历强行推进，而是按验证门槛推进。
- 每一期只回答一个核心问题：先验证交易是否真实，再验证供给是否可放大，再验证经营工具是否值得重投入，最后再扩展多端入口和复杂商单。
- V1 之前必须允许人工运营、表格台账和轻量下单方式参与验证；不要因为未来要自动化，就提前建设完整后台。
- 多端扩展只改变入口、展示和归因，不拆分商品库、开团、订单、售后或收益账本。
- 支付分账、退款回退、真实自动打款未核验前，收益只表达账本状态，不承诺自动结算。

## 2. 多期路线总览

| 版本 | 阶段目标 | 功能边界 | 核心验证指标 | 进入下一阶段条件 | 前置/后置能力 |
| --- | --- | --- | --- | --- | --- |
| V0 人工验证期 | 证明“社群信任关系能转化为真实订单，且履约/售后/收益解释能被接受” | 不做完整产品；使用人工货盘、表格台账、微信群、小程序/原型或轻量下单方式验证活动、卡券和低风险团购 | 有效团长访谈数、可售供给数、真实支付订单数、核销/履约完成率、退款/投诉原因、团长复盘意愿 | 至少形成可复用的首批供给规则、团长路径、订单台账和售后处理方式；明确 V1 必做/不做边界 | 前置：供给白名单、团长样本、人工订单台账。后置：自动选品、自动结算、多端入口 |
| V1 交易闭环期 | 用同一微信小程序跑通 C 端下单与 B 端经营，帮助新手完成第一团 | 社群建店、商品库/供给库、开团发布、活动报名、卡券团购、低风险普通团购、订单、核销/履约、售后、收益账本、一级推广 | 新手建店完成率、首单开团完成率、支付订单数、核销率、售后工单率、来源可追踪订单占比、团长二次开团率 | 活动/卡券/低风险团购至少有一个场景跑出稳定复用路径；售后和收益解释不再主要依赖私聊补救 | 前置：V0 供给与流程验证。后置：完整选品中心、团长 App、小红书/抖音消费端、内容课程独立体系 |
| V2 供给放大期 | 让团长从“能开第一团”进入“能持续选品和复用供给” | 选品中心、审核货盘分层、社群推荐供给包、素材包、人工推荐理由、低风险商品/自提/预售/到店服务受控扩展 | 供给复用率、推荐供给开团率、二次开团率、推广订单占比、退款率、供给风险暂停数 | 审核货盘和推荐逻辑能支撑多社群复用；新增场景未显著拉高售后和履约风险 | 前置：V1 交易闭环和售后规则。后置：数据驱动推荐、重度经营 App、规模化本地服务 |
| V3 经营效率期 | 为高频团长和成熟经营者提供更专业的经营工具 | 团长 App 或专业 B 端工具、经营数据、复购运营、素材管理、订单售后批处理、收益分析、选品中心深度使用 | 高频团长占比、月均开团数、订单处理时长、复购率、经营数据使用率、App/专业工具留存 | B 端高频经营需求明确，微信小程序经营台已经成为效率瓶颈，且有足够团长样本愿意迁移 | 前置：V2 供给与经营数据沉淀。后置：多端 C 端规模化投放、复杂商单和内容课程体系 |
| V4 多端与复杂交易期 | 扩展消费入口和更复杂的变现形态，但共用统一交易底座 | 微信/小红书/抖音小程序消费端、多平台归因、复杂商单、内容/课程受控验证、更多本地生活场景 | 多平台来源订单占比、跨端归因完整率、商单履约完成率、内容/课程退款率、平台服务费贡献 | 主交易底座稳定，多端带来新增订单而不是运营混乱；复杂交易的资质、履约、售后和结算责任已确认 | 前置：统一交易底座、售后规则、支付/退款核验。后置：多级分销、强监管品类、强效果承诺业务 |

## 3. 阶段门槛说明

V0 进入 V1 的条件：

- 已确认首批验证城市、团长样本和低风险供给范围。
- 已跑通至少一种真实支付或准真实支付订单路径，并记录履约、退款和售后问题。
- 已形成 V1 必做清单和明确排除项，避免把人工验证中所有临时动作都产品化。

V1 进入 V2 的条件：

- 活动报名、卡券团购或低风险普通团购中至少一个场景具备可复用开团模板。
- 新手群主的建店、选品、发布、邀请、首单、复盘路径不再完全依赖人工一对一带教。
- 订单、核销/履约、售后和收益账本能解释主要交易争议。

V2 进入 V3 的条件：

- 选品中心和推荐供给包被团长持续使用，而不是只在运营强推时使用。
- 成熟团长在订单、售后、素材、数据和复购上出现明显高频工作量。
- 团长 App 或专业 B 端工具能解决真实效率瓶颈，而不是为了多端而多端。

V3 进入 V4 的条件：

- 统一交易底座稳定，新增入口不会导致货盘、订单、售后和结算重复建设。
- 支付分账、退款回退、结算阻断和平台责任边界已经核验。
- 小红书/抖音等消费入口有明确订单增量假设，复杂商单或内容课程已有资质、履约和售后规则。

## 4. 产品方向变化对路线的影响

| 方向变化 | 路线影响 |
| --- | --- |
| 若新手群主验证更强 | 加强 V0/V1 的冷启动路径、首单清单和人工带教模板；不提前做 App 和复杂选品自动化 |
| 若成熟社群经营者验证更强 | V1 保持交易闭环，V2 可提前强化选品中心、批量开团、订单售后和经营数据 |
| 若活动/卡券显著优于电商 | 延长活动和卡券深挖周期，电商继续作为低风险对照，不急于扩品类 |
| 若本地服务需求强但售后复杂 | 先补履约规则、预约改期、投诉和责任边界，再扩大服务供给 |
| 若内容/课程需求出现 | 先以体验课、工作坊、活动或卡券形态验证，不直接进入独立课程体系、讲师分账和强效果承诺 |
| 若公域内容平台带来明显订单 | 仍放在 V4 或 V3 后验证消费入口，不改变 V1 同一微信小程序 C/B 双通道的首发承载 |
| 若支付/退款能力核验不通过 | 自动打款继续后置，V1/V2 只保留收益账本、售后冻结、退款冲回和人工结算说明 |

## 5. 当前待确认点

- V0 验证样本、真实订单数量和停止/进入 V1 的最低门槛。
- V1 进入 V2 时，活动、卡券、普通团购三类场景分别需要达到哪些可复用指标。
- 团长 App 的启动门槛是否只看高频团长规模，还是还要看订单售后处理时长和经营数据使用率。
- 小红书/抖音消费端优先级，是否由订单增量、获客成本、履约风险和内容生产能力共同决定。
