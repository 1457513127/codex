# 和趣项目协作约束

本文件是所有后续开发和策划会话的协作约束。开始修改前必须阅读：

1. `README.md`
2. `01-研究与规划/当前产品策划.md`
3. `00-项目治理/项目协作手册.md`
4. `00-项目治理/决策与问题台账.md`
5. `docs/UPDATE_LOG.md`
6. `design-system/README.md`（涉及界面时）

## 长期上下文

- 不依赖单一 Codex 对话记忆；项目事实以仓库文件为准。
- 当前产品口径只写入 `01-研究与规划/当前产品策划.md`。
- 重大决策和未确认问题只写入 `00-项目治理/决策与问题台账.md`。
- 过程记录只写入 `docs/UPDATE_LOG.md`。
- 历史材料只作为追溯依据，不直接代表当前 MVP、会议口径或开发范围。

## 策划规则

- 当前产品策划尚未最终定稿，先按策划成熟度推进。
- 不要过早把商品库、订单、分销、结算等未确认功能写成最终规格。
- 默认遵循“先边界、再场景、再版本、再功能”：边界不清时不写 PRD，场景不稳时不改原型，版本位置不明时不扩功能。
- 仅战略和规划讨论不自动修改 PPT 或原型；只有形成明确版本口径且用户要求更新交付物时才修改。

## 小范围更新

- 一次更新只解决一个清晰目标；默认不混入无关重构、重命名或视觉翻新。
- 默认变更预算：不超过 5 个既有业务/界面文件，不超过 200 行既有代码变更。文档结构整理可作为单独目标执行，但不得顺带改业务口径。
- 超出范围前必须先解释原因、列出额外文件，并取得用户确认。
- 提交前列出实际改动文件和验证结果。

## 界面一致性

- 所有界面必须使用 `design-system/airbnb-tokens.css`，不得自行新增品牌色、阴影或字体栈。
- 设计依据为 `design-system/README.md` 引用的 Airbnb inspired `DESIGN.md`。
- 新组件先复用已定义 token；确需新增 token 时，先补充设计规范及更新日志。

## Notion 记录

- 每次代码、文档或交付物更新都必须在项目 Notion 更新文档追加记录：目标、改动文件、设计影响、验证、提交链接和待办。
- 若当前会话未连接 Notion，先写入 `docs/UPDATE_LOG.md` 并标为 `待同步`，连接可用后优先补同步记录。

## 默认发布

- 用户提出可执行的代码、文档或界面更新后，完成并验证通过时，默认更新本地日志与 Notion、创建聚焦提交并推送 GitHub。
- 仅讨论、研究、规划或尚未完成/验证失败的工作不自动发布。
- 用户明确要求仅本地修改、暂不提交、暂不推送或等待确认时，以该次要求为准。
