# 项目协作手册

本文只说明和趣项目如何长期协作，不承载产品业务口径。产品口径统一读取 [当前产品策划](../01-研究与规划/当前产品策划.md)。

## 1. 工作方式

采用：

`一个项目仓库 + 多个专题对话 + 少量权威文档 + 历史归档`

- Codex 中继续使用当前 `New project`，不要为产品、PPT、原型拆多个项目。
- 新对话先读：`README.md`、`01-研究与规划/当前产品策划.md`、本手册、`00-项目治理/决策与问题台账.md`。
- 项目事实源是仓库文件，不是聊天记录。
- 历史资料归档不删除，但归档内容不作为当前口径。

## 2. 对话分层

当前产品尚未定稿，先按策划成熟度分层：

- `00-项目总控与长期记忆`
- `01-产品定位与用户分层`
- `02-市场赛道与竞品研究`
- `03-变现场景与业务主线`
- `04-产品版本与多期路线`
- `05-MVP范围与不做清单`
- `06-功能方案沉淀区`
- `07-PPT与会议材料`
- `08-原型与界面走查`

等 MVP 范围稳定后，再从 `06-功能方案沉淀区` 拆出商品库、订单、分销、结算等具体 PRD 专题。

## 3. 策划方法

默认遵循：

`先边界 -> 再场景 -> 再版本 -> 再功能 -> 再交付物`

- 边界不清时，不写 PRD。
- 场景不稳时，不改原型。
- 版本位置不明时，不扩功能。
- PPT 和原型只表达已确认口径，不替产品拍板。
- AI 输出必须可见、可改、可控、可回溯。

## 4. 文件落盘规则

| 信息类型 | 写入位置 |
| --- | --- |
| 当前产品口径 | `01-研究与规划/当前产品策划.md` |
| 关键决策和待确认问题 | `00-项目治理/决策与问题台账.md` |
| 每次有效改动 | `docs/UPDATE_LOG.md` |
| PPT/主讲材料 | `02-项目会/` |
| 原型/走查脚本 | `03-演示界面/` |
| 历史材料 | `99-历史归档/` |

## 5. 小范围更新规则

- 一次更新只解决一个清晰目标。
- 默认不顺手修改 PPT、原型或无关文档。
- 超出 5 个既有业务/界面文件或明显跨主题时，先说明范围。
- 涉及界面时必须使用 `design-system/airbnb-tokens.css`。

## 6. 发布规则

完成有效文档、代码或交付物更新后：

1. 更新 `docs/UPDATE_LOG.md`。
2. 同步 Notion；若连接失败，在日志中标记待同步。
3. 创建聚焦 GitHub 提交并推送 `master`。
