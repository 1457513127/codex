# 项目协作手册

本文是和趣项目长期策划、专题讨论和交付协作的第一入口。

## 1. 工作方式

和趣项目采用：

`一个项目仓库 + 多个专题对话 + 文件化长期记忆`

- 继续使用当前 Codex 项目 `New project`，不要为产品、PPT、原型分别创建多个 Codex 项目。
- 在同一个项目下按专题新建多个对话，不把全部复杂讨论塞进一个长对话。
- 项目事实源是仓库文件，不是某一次聊天记录。
- Notion 用于外部协作记录，GitHub 仓库用于保存最终项目资产和历史追踪。

推荐专题对话命名：

- `00-项目总控与长期记忆`
- `01-产品定位与用户分层`
- `02-市场赛道与竞品研究`
- `03-变现场景与业务主线`
- `04-产品版本与多期路线`
- `05-MVP范围与不做清单`
- `06-功能方案沉淀区`
- `07-PPT与会议材料`
- `08-原型与界面走查`

当前产品策划尚未最终定稿，专题先按“策划成熟度”划分，不提前把商品库、订单、分销、结算等功能拆成固定孤岛。等产品方向和 MVP 范围稳定后，再从 `06-功能方案沉淀区` 拆出具体 PRD 专题。

## 2. 产品策划方法论

和趣长期策划遵循“先边界、再场景、再版本、再功能”的工作法：

1. 先锁边界：每轮讨论先回答产品解决什么、不解决什么、第一版核心动作是什么。
2. 再看场景：所有决策优先回到真实用户场景、交易动作、履约售后和收益解释，不以主观感觉或单页 UI 为依据。
3. 再排版本：只让每个版本验证 1-3 个关键假设，明确本期验证什么、不验证什么。
4. 再沉淀功能：商品库、开团、订单、分销、选品中心等功能先进入沉淀区，待方向稳定后再拆 PRD。
5. 最后做交付物：PPT 和原型只表达已确认口径，不承担替产品拍板的职责。

AI 辅助能力只能作为草稿和提效工具，必须满足“可见、可改、可控、可回溯”：原始依据可见、输出可编辑、用户确认后生效、每次关键变化能回到决策或更新记录。

## 3. 新对话读取顺序

每个新 Codex 对话开始时，先读：

1. `README.md`
2. `README-项目上下文总览.md`
3. `00-项目治理/项目协作手册.md`
4. `00-项目治理/决策与问题台账.md`
5. `docs/PROJECT_MEMORY.md`
6. `docs/CHANGE_POLICY.md`
7. `docs/UPDATE_LOG.md`
8. `design-system/README.md`（涉及界面时）

## 4. 信息落盘规则

| 信息类型 | 写入位置 | 示例 |
| --- | --- | --- |
| 稳定结论 | `docs/PROJECT_MEMORY.md` | 产品定位、首发边界、分销规则、设计系统约束 |
| 重大决策 | `00-项目治理/决策与问题台账.md` | 为什么采用商品库 + 开团类型 |
| 待确认问题 | `00-项目治理/决策与问题台账.md` | 支付主体、首城范围、供给审核责任 |
| 过程记录 | `docs/UPDATE_LOG.md` | 每次改了什么、验证了什么、是否同步 Notion |
| 专题材料 | 对应专题文档 | PRD、货盘规则、会议材料、原型说明 |

没有落盘的聊天内容不视为项目事实。

## 5. 每轮工作分类

| 类型 | 产出 | 是否默认改 PPT/原型 |
| --- | --- | --- |
| 战略定位 | 方向判断、边界、决策 | 否 |
| 功能规划 | 蓝图、模块说明、优先级 | 否 |
| PRD 拆解 | 页面、对象、流程、验收标准 | 否 |
| 原型/PPT | 可点击原型、演示稿、走查脚本 | 是，仅限用户明确要求 |
| 市场研究 | 调研结论、证据边界、引用来源 | 否 |
| 执行任务 | 文件修改、验证、提交、推送 | 按任务范围 |

仅讨论和规划不自动修改 PPT/原型。只有形成明确版本口径，且用户要求更新交付物时，才进入 PPT 或原型修改。

## 6. 交付流程

一次有效交付必须包含：

1. 读取治理入口、上下文总览和项目记忆。
2. 明确本轮目标和不做范围。
3. 小范围修改必要文件。
4. 执行适合本轮的验证。
5. 更新 `docs/UPDATE_LOG.md`。
6. 同步 Notion。
7. 创建聚焦 GitHub 提交并推送。

## 7. 长期维护习惯

- 不把所有历史塞进 README。
- 不把临时讨论写入长期记忆。
- 不把尚未确认的设想写成已决策。
- 不让 PPT 和原型超前于产品蓝图。
- 不为每个平台、每个话题复制项目目录。
- 历史资料采用归档不删除，当前入口只保留权威文件。
