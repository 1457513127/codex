# 上下文记忆机制

本项目的长期记忆通过文件维护，目标是让任何新 Codex 对话都能恢复上下文。

## 1. 四类信息边界

| 信息类型 | 写入位置 | 示例 |
| --- | --- | --- |
| 稳定结论 | `docs/PROJECT_MEMORY.md` | 产品定位、首发边界、分销规则、设计系统约束 |
| 重大决策 | `00-项目治理/决策记录.md` | 为什么采用商品库 + 开团类型，为什么双通道不放项目标题 |
| 待确认问题 | `00-项目治理/待确认问题.md` | 支付主体、结算方式、首城范围、首批供给白名单 |
| 过程记录 | `docs/UPDATE_LOG.md` | 每次改了什么、验证了什么、是否同步 Notion 和 GitHub |

## 2. 什么进入长期记忆

写入 `docs/PROJECT_MEMORY.md` 的内容必须满足至少一个条件：

- 会影响后续所有讨论或实现。
- 已被用户明确确认。
- 属于不可轻易改变的项目约束。
- 新对话如果不知道会明显做错。

不进入长期记忆的内容：

- 临时想法。
- 尚未确认的猜测。
- 某次会议的过程性表达。
- 单次 PPT 或原型的实现细节。

## 3. 新会话恢复上下文

新会话必须按顺序读取：

1. `00-项目治理/项目工作方式.md`
2. `00-项目治理/专题索引.md`
3. `README-项目上下文总览.md`
4. `docs/PROJECT_MEMORY.md`
5. `docs/CHANGE_POLICY.md`
6. `docs/UPDATE_LOG.md`
7. `design-system/README.md`（涉及界面时）

如果用户提出新方向，先判断它是稳定决策、待确认问题，还是仅属于专题讨论。

## 4. 对话结束沉淀规则

专题对话结束时：

- 如果形成新决策，补入决策记录。
- 如果产生未决问题，补入待确认问题。
- 如果改变稳定结论，更新项目记忆。
- 如果修改交付物，更新更新日志并同步 Notion。

没有落盘的聊天内容不视为项目事实。
