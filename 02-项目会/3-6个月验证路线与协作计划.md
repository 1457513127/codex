# 和趣亲子活动平台：3-6 个月验证路线与协作计划

版本：V1.0  
日期：2026-05-26  
规划原则：前三个月验证分销闭环是否成立，后三个月只在门槛通过后扩量，不以功能数量替代业务验证。

## 1. 核心验证命题

> 通过团长轻主页和受控一级帮卖，公司能否获得可归因的新增亲子活动订单，且退款/投诉质量不明显变差、扣除佣金和售后成本后仍存在正贡献毛利？

## 2. 六个月阶段图

| 阶段 | 时间 | 主要目标 | 可交付物 | 继续门槛 |
| --- | --- | --- | --- | --- |
| Phase 0：可行性确认 | 第 1-2 周 | 确认口径、支付、角色与试点资源 | 数据字典、支付结算结论、首批名单、范围确认 | 不存在资金/责任硬阻断 |
| Phase 1：MVP 研发与内测 | 第 3-8 周 | 建成主页、归因、推广订单、佣金账本和运营审核 | 内测小程序/原型、埋点、后台导出、测试报告 | 核心链路可追踪且权限正确 |
| Phase 2：真实受控试点 | 第 9-12 周 | 以真实收费订单验证帮卖链路 | 10 名团长、20 场活动、周报、售后复盘 | 指标和护栏达到门槛 |
| Phase 3：扩量或转向 | 第 4-6 月 | 扩亲子供给/团长或收缩方案 | 渠道比较、主办方质量层级、版本二需求 | 增量和正贡献毛利成立 |

## 3. Phase 0：第 1-2 周，可行性确认

### 3.1 运营任务

| 任务 | 输出 | 负责人 | 完成标记 |
| --- | --- | --- | --- |
| 梳理现有帮卖团长 | 至少 10 名候选团长、社群类型/规模档位、合作意愿 | 运营 | 待指定 |
| 筛选活动 | 至少 20 场候选活动，标记主办方、退款风险、是否可推广 | 运营 | 待指定 |
| 复盘退款 | 高退款活动原因、场次取消/规则/履约分类 | 运营 | 待指定 |
| 整理售前问题 | 家长常问问题及现有回答方式 | 运营 | 待指定 |

### 3.2 产品/数据任务

| 任务 | 输出 | 负责人 | 完成标记 |
| --- | --- | --- | --- |
| 对账源表口径 | `实际收入`、退款金额、订单数的正式定义 | 产品 + 财务/运营 | 待指定 |
| 定义新增字段 | 主办方、团长、归因、佣金、成本、退款原因等字段表 | 产品 + 开发 | 待指定 |
| 确认试点指标 | 周报模板与停止规则 | 产品 | 待指定 |

### 3.3 技术/支付任务

| 任务 | 输出 | 负责人 | 完成标记 |
| --- | --- | --- | --- |
| 角色模型评估 | 主办方/团长双身份及权限改造粗估 | 开发 | 待指定 |
| 归因实现评估 | 小程序分享参数、归因锁定与冲突规则 | 开发 | 待指定 |
| 佣金状态评估 | 状态机、订单退款联动、账单导出 | 开发 | 待指定 |
| 支付路径确认 | 团长主体类型、结算、退款后回退/冲抵可执行方案 | 支付/法务 | 待指定 |

### 3.4 Phase 0 Go/No-Go

| 判断 | Go | No-Go / 转向 |
| --- | --- | --- |
| 支付路径 | 可形成合规延迟结算和退款处理路径 | 佣金暂不真实结算，只验证归因和账本 |
| 试点资源 | 有 10 名团长和 20 场活动候选 | 缩小为手工试点或继续招募 |
| 数据能力 | 关键归因与退款字段可开发/导出 | 暂不扩量 |

## 4. Phase 1：第 3-8 周，MVP 研发与内测

### 4.1 开发范围

| Epic | 功能 | 首轮必要性 |
| --- | --- | --- |
| 数据底座 | 主办方/团长 ID、归因入口、佣金状态、退款原因、成本字段、事件埋点 | P0 |
| 团长主页 | 身份、推荐关系披露、可报名活动货架、分享归因 | P0 |
| 帮卖管理 | 可推广活动、上架主页、专属分享、推广订单 | P0 |
| 佣金账本 | 预估、待生效、可结算、已结算、已回退 | P0 |
| 运营后台 | 活动允许/暂停分销、团长审核、归因与退款导出 | P0 |
| 家长详情增补 | 展示团长、实际主办方、平台售后关系 | P0 |
| 分析看板 | 基础漏斗和周报导出即可 | P1 |

### 4.2 明确排除

- 商品、物流、库存、优惠券。
- 多级分销和招募奖励。
- 主页动态、评论或群成员社区。
- 复杂推荐算法或 AI 选品。

### 4.3 内测验收

| 测试项 | 通过标准 |
| --- | --- |
| 角色与权限 | 团长不能修改主办方活动或看到儿童敏感明细 |
| 归因 | 主页和活动分享支付订单可记录唯一团长与来源 |
| 责任展示 | 家长付款前能清楚看到主办方、团长和平台售后边界 |
| 佣金 | 支付、核销、完成、退款可驱动预设状态变化 |
| 高风险治理 | 运营可暂停某活动的分销并即时从货架移除 |
| 数据 | 周报所需字段可导出，核心链路无缺失 |

## 5. Phase 2：第 9-12 周，真实受控试点

### 5.1 试点配置

| 项目 | 目标 |
| --- | ---: |
| 审核团长 | 至少 10 名 |
| 上架可分销活动 | 至少 20 场 |
| 优先活动类型 | 标准化票务、低退款室内亲子体验 |
| 受限活动类型 | 户外、研学、采摘及既往高退款活动 |
| 试点周期 | 连续 4 周 |

### 5.2 每周必看数据

| 维度 | 数据 |
| --- | --- |
| 渠道漏斗 | 主页访问、活动点击、下单、支付、核销、退款 |
| 团长表现 | 分享、归因订单、支付 GMV、退款、佣金、重复推广 |
| 活动质量 | 主办方、取消、退款、核销、投诉、保险异常 |
| 经济性 | 售价、结算/采购成本、支付成本、佣金、售后成本、贡献毛利 |
| 合规护栏 | 权限异常、责任展示错误、儿童信息/保险问题、支付问题 |

### 5.3 通过门槛

| 指标 | 第 3 月评审门槛 |
| --- | ---: |
| 订单归因、退款、佣金状态完整率 | 100% |
| 帮卖成交订单可追踪至主页/专属分享 | 至少 80% |
| 严重隐私、资金、履约责任展示事故 | 0 |
| 团长持续推广 | 有可识别的重复分享和重复成交团长 |
| 渠道质量 | 退款/投诉不明显恶化，差异需有原因解释 |
| 毛利判断 | 已具备完整成本口径，可判断是否正贡献毛利 |

## 6. Phase 3：第 4-6 月，扩量或转向

### 6.1 若试点通过

| 扩展项 | 目标 |
| --- | --- |
| 团长扩展 | 扩大亲子群团长数量，但保留审核机制 |
| 外部供给扩展 | 引入更多质量可控的亲子主办方 |
| 数据增强 | 团长经营数据、结算批次、主办方质量分层 |
| 主页优化 | 增加轻量筛选、历史报名入口或收藏/复访能力，以数据决定 |

### 6.2 若试点不通过

| 问题 | 转向方案 |
| --- | --- |
| 主页没有转化价值 | 缩为活动专属分享和归因，不继续做主页运营 |
| 帮卖不产生增量 | 回到自有社群运营，暂停分销扩量 |
| 退款或售后失控 | 收缩活动白名单，优先解决供给质量 |
| 资金结算不可行 | 先保留推荐归因，不上线自动佣金 |

### 6.3 不自动进入的方向

商品团购、成人体验和其他本地服务均不能作为“第 4 月默认开发项”，应在亲子分销数据明确后分别立项。

## 7. 团队分工建议

| 团队 | 负责结果 | 不能只交付 |
| --- | --- | --- |
| 产品 | 范围、角色、规则、指标、演示与评审结论 | 功能清单 |
| 运营 | 团长和供给名单、退款原因、活动质量治理、试点执行 | “团长说想要” |
| 开发 | 权限、归因、佣金、导出、埋点、技术风险与排期 | 页面完成截图 |
| 财务/支付/法务 | 佣金结算和退款路径、协议/披露边界 | 口头认可 |
| 负责人 | Go/No-Go、资源与扩量条件 | 无门槛的上线要求 |

## 8. 评审节奏

| 节点 | 会议内容 | 参与人 |
| --- | --- | --- |
| 启动会 | 确认 Phase 0、MVP、责任人 | 全体 |
| 第 2 周门槛会 | 数据口径、支付结算、首批名单、技术粗估 | 产品/运营/开发/支付 |
| 第 6 周演示会 | MVP 可走查、埋点和权限验收 | 产品/开发/运营 |
| 第 8 周上线评审 | 真实试点开关与活动白名单 | 全体 |
| 第 12 周阶段评审 | 是否扩量、收缩或暂停 | 负责人及核心团队 |
| 第 6 月战略复盘 | 是否进入新场景独立评估 | 管理层/产品/运营 |
