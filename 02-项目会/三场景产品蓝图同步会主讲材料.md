# 社群店铺驱动三场景产品蓝图同步会主讲材料

版本：V2.2

日期：2026-05-27

会议性质：产品经理同步产品功能规划、首发产品形态与迭代路线

权威规划：[社群店铺驱动的三场景私域变现平台产品蓝图](../01-研究与规划/三场景私域变现平台产品蓝图.md)

## 1. 本次同步要讲清什么

本次会议不是现场重新讨论方向，也不以开发如何预研为主线。产品侧要讲清：

> 和趣不是先做一个大而全的社群平台，而是让团长通过社群店铺完成活动、电商与本地服务交易，并把每一笔订单的履约、售后、退款、核销和收益结算变得可监督。

会后团队需要形成统一理解：

1. 产品主线：社群建店、发布/选卖供给、分享成交、订单服务台、履约售后、一级收益结算。
2. 首发优先级：活动变现为主验证场景，电商变现为对照验证场景，本地服务为受控补充场景。
3. 产品形态：团长端、团员端、创建活动、创建商品、详情页、购买须知、订单详情和售后服务台必须连成闭环。
4. 功能边界：首发做交易履约闭环和一级收益，不做多级分销、保证金账户、会员权限、广告或 AI 自动发布。

## 2. 新资料给出的纠偏

| 来源 | 应采纳的产品判断 | 首发不承诺 |
| --- | --- | --- |
| `新录音.pdf` | 平台价值来自服务、售后和交付约束；“活动小群”应转译为订单服务台与双客服响应 | 大群 IM、内容社区、复杂群互动 |
| `商单系统功能及抽成规则讨论.pdf` | 谁借助平台赚到钱，平台才有抽佣依据；加价、佣金、优惠和退款回退必须算清 | 多级转单、经销商层级、保证金、会员可见 |
| 产品落地稿 | 已经出现团长端、团员端、创建活动、创建商品、详情页、订单详情等 MVP 页面骨架 | 逐像素复刻视觉或声明后端已可用 |

一句话讲法：和趣的护城河不是“我有货让你卖”，而是“你卖出去以后，平台能约束履约、售后和结算”。

## 3. 小红书调研给出的方向信号

本轮市场调研读取 `100` 篇有效详情、`467` 条首屏可见评论，并整理 `10` 个官方公开竞品来源。调研用于判断优先级，不作为营收或市场规模证明。

| 场景 | 有效详情 | 首屏评论 | 付费线索 | 方向分 | 产品判断 |
| --- | ---: | ---: | ---: | ---: | --- |
| 活动变现 | 20 | 99 | 8 | 106.3 | 主验证场景 |
| 电商变现 | 20 | 146 | 4 | 103.2 | 对照验证场景 |
| 本地服务变现 | 20 | 59 | 1 | 70.1 | 受控补充场景 |
| 会员订阅变现 | 20 | 71 | 5 | 81.1 | 后续经营能力线索 |
| 知识/内容服务变现 | 20 | 92 | 3 | 83.3 | 后续行业扩展线索 |

产品解读：

- 活动变现与现有和趣活动交易基础最接近，且调研中付费线索最多，适合成为主验证场景。
- 电商变现评论更多、竞品能力更密集，但履约与售后复杂度更高，适合做对照验证而不是压过活动。
- 本地服务调研得分较低，只保留到店预约核销型低风险服务。
- 会员订阅和知识/内容服务不单列为首发交易场景，可在后续作为复购、权益或行业扩展能力评估。

## 4. 证据边界必须说清

| 证据 | 可以说明 | 不能说明 |
| --- | --- | --- |
| 小红书帖子与评论 | 用户表达过哪些需求、痛点和场景热度 | 真实营收、市场规模、付费率、转化率或 ROI |
| 官方竞品页面 | 市场上已有店铺、预约、核销、会员、分销等能力教育 | 目标团长实际采用率或我们应照搬其产品形态 |
| 现有和趣活动数据 | 活动交易和售后有真实基础 | 帮卖、社群归因、电商或本地服务已经被验证 |
| 商单/录音讨论 | 可作为内部产品假设和规则来源 | 不能当作复杂商单能力已验证或必须首发 |

因此，本次产品路线采用“证据支持优先级，真实交易验证商业成立”的原则。

## 5. 产品主线

| 环节 | 产品功能 | 关键判断 |
| --- | --- | --- |
| 社群建店 | 选择社群主类型和兴趣标签，生成专属店铺模板 | 让团长先有可经营的私域承接入口 |
| 发布/选卖供给 | 活动、商品、到店服务统一进入审核供给库 | 先控制履约责任与售后风险 |
| 分享成交 | 店铺、单品、优惠券分享，记录唯一直接推广方 | 一级归因是收益结算基础 |
| 订单服务台 | 咨询、官方客服、发布方客服、售后工单、退款进度 | 平台价值从交易后开始变得不可替代 |
| 履约售后 | 活动核销、物流/自提/权益交付、到店服务核销 | 三场景共用订单与售后底座 |
| 收益经营 | 固定佣金、加价帮卖、优惠承担、退款冲回 | 先清晰展示收益，再核验真实自动结算 |

## 6. 首发 MVP 页面

| 页面 | 首发信息 | 说明 |
| --- | --- | --- |
| 团长端主页 | 我的收入、订单/报名管理、核销管理、社群设置、邀请帮卖、客服 | 团长经营入口 |
| 团员端主页 | 活动订单、卡券订单、物流订单、自提订单、浏览历史 | 消费者履约入口 |
| 创建活动 | 活动信息、地点、场次、票种/分组、报名信息、保险、退款规则、核销方式 | 活动主验证核心 |
| 创建商品 | 商品信息、售卖时间、售卖方式、售后规则、规格、库存、限购、购买需填信息 | 电商对照验证核心 |
| 详情与购买须知 | 价格、倒计时、履约信息、详情、退改/售后规则、咨询、立即购买 | 转化和规则披露入口 |
| 订单详情 | 支付状态、待入账、金额、报名/收货/保险信息、退款入口 | 售后和结算依据 |
| 订单服务台 | 双客服响应、售后工单、退款进度、结算阻断 | 平台护城河页面 |

## 7. 首发三场景策略

| 场景 | 首发定位 | 首发功能 | 暂不开放 |
| --- | --- | --- | --- |
| 活动变现 | 主验证场景 | 类型、场次、票种、报名、保险、支付、核销、退改 | 高风险户外、复杂多日行程、强效果承诺活动 |
| 电商变现 | 对照验证场景 | 规格、库存、物流、自提、虚拟权益、退换规则 | 食品生鲜、医疗美容、资质复杂或售后责任不清品类 |
| 本地服务变现 | 受控补充场景 | 门店、预约时段、服务规格、到店核销、取消/投诉 | 上门服务、医疗美容、强效果或人身安全责任较高服务 |

## 8. 活动变现首发功能

活动是主验证场景，优先证明从社群触达到报名核销、退款售后的闭环。

| 模块 | 功能要求 |
| --- | --- |
| 活动发布 | 活动名称、描述、图片、类型、地点、主办/履约方 |
| 场次票种 | 多场次、票种/分组、名额、价格、报名截止时间 |
| 报名信息 | 昵称、手机号、车牌、保险信息等可配置字段 |
| 购买须知 | 不同场次可配置退款规则和注意事项 |
| 核销售后 | 现场核销、活动取消、全额/部分退款、投诉工单 |

## 9. 电商变现首发功能

电商是对照验证场景，重点不是“商品列表”，而是三种履约方式是否能共用同一交易底座。

| 履约方式 | 首发能力 | 关键风险 |
| --- | --- | --- |
| 物流商品 | 规格、库存、发货状态、物流信息、退货退款 | 发货责任与售后周期 |
| 自提商品 | 提货点、提货时间、提货码、未核销退款 | 库存锁定与核销凭证 |
| 虚拟权益 | 电子券/次卡发放、使用状态、未使用退款 | 权益规则披露与过期争议 |

## 10. 本地服务首发功能

本地服务只做受控补充，首发限定为到店预约核销型供给。

| 模块 | 功能要求 |
| --- | --- |
| 服务上架 | 门店、服务规格、可预约时段、服务方、取消规则 |
| 预约下单 | 消费者选择时段并支付，订单记录服务方与推荐来源 |
| 到店核销 | 到店完成服务后核销，保留核销凭证 |
| 取消投诉 | 支持取消、改期、未服务退款和投诉工单 |
| 风险控制 | 不开放上门、医疗美容、强效果承诺或安全责任高的服务 |

## 11. 增长与收益功能

| 功能 | 首发规则 |
| --- | --- |
| 专属分享 | 店铺、单品和优惠券入口均记录唯一直接推广团长 |
| 固定佣金分享赚 | 发布方设置固定佣金，消费者价格一致 |
| 加价帮卖 | 发布方设置底价，推广团长设置终端售价 |
| 模式互斥 | 同一供给只能选择固定佣金或加价帮卖，一单只有一名直接推广方 |
| 优惠券 | 支持店铺券和单品券；推广方发券优先从推广收益承担 |
| 退款回退 | 全额退款全链取消；部分退款按比例冲回收益与优惠 |
| 自动结算 | 作为目标能力保留，支付与退款回退未核验前不承诺真实打款 |

## 12. 产品迭代路线

| 阶段 | 产品目标 | 功能重点 | 验证指标 |
| --- | --- | --- | --- |
| A：交易履约闭环 | 团长和团员能走通发布、购买、服务、退款和收益 | 双端主页、发布活动/商品、详情、订单、订单服务台 | 主线可演示、规则口径一致 |
| B：场景验证 | 验证三场景优先级是否成立 | 活动票务、电商三履约、到店预约、优惠券、一级收益 | 真实订单、来源可追踪、售后可闭环 |
| C：经营效率增强 | 提升复购和分享效率 | 店铺数据、复购触达、素材提示、优惠策略、风险暂停 | 复购、退款率、售后成本和团长持续经营 |
| D：扩展判断 | 对被验证的方向加深 | AI 话术、群接龙、会员经营、商单扩展、内容服务或更多品类另案验证 | 新能力能提升成交效率且不放大履约风险 |

## 13. 首发不做什么

- 不做多级分销、发展下级、层层转单或二级返佣。
- 不做保证金账户、会员分层可见、经销商权限和复杂社群角色体系。
- 不做自动选货、自动上架或自动对外发布营销内容。
- 不把 AI 朋友圈/社群/小红书/抖音话术生成和群接龙作为首发核心。
- 不开放食品生鲜、上门服务、医疗美容、复杂内容课程、直播训练营或广告变现。
- 不在支付与退款回退核验完成前宣称自动分账、即时佣金到账或资金保障已经可用。

## 14. 现场结论表达

- 和趣当前要做的是社群店铺驱动的三场景私域变现平台，但核心竞争力落在订单服务、履约售后和结算监督。
- 小红书调研支持活动和电商优先验证；活动为主验证，电商为对照，本地服务受控补充。
- 商单讨论提供了收益规则方向，但复杂商单系统不进入首发。
- 首发核心不是功能堆叠，而是团长端、团员端、供给发布、购买须知、订单详情、售后服务台和一级收益闭环。
