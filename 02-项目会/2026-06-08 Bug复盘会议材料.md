# 2026-06-08 Bug 复盘会议材料

适用范围：仅复盘飞书《团小助测试bug整理》中 `v1.1`、`v1.2`、`v1.3` 三个视图。

会议目标：把 bug 文档从“问题清单”复盘成“下轮减少重复问题的协作机制”。本次不逐条分派 bug，而是对齐问题趋势、根因、回归矩阵和后续协作动作。

## 1. 先抛出的主结论

- `v1.1` 是基础体验和 UI 一致性问题集中暴露：输入、文案、按钮、图标、页面滑动、上传图片、基础发布限制等。
- `v1.2` 转向交易链路问题：团购发布、规格、限购、订单详情、报名信息、退款、物流字段等。
- `v1.3` 转向复杂状态问题：自提、卡券、售后、退款、活动状态、订单状态、售卖结束、重新发布等。
- 质量风险已经从“页面做没做好”升级为“交易、履约、售后状态是否定义清楚、实现一致、回归完整”。

建议现场避免把结论表达成“bug 太多”，而是表达为：随着业务闭环变复杂，问题从视觉细节自然迁移到状态组合和边界规则，下一版要把产品规则、开发自测和回归验收前置。

## 2. 60 分钟会议议程

| 时间 | 议题 | 讨论重点 | 产出 |
| --- | --- | --- | --- |
| 5 分钟 | 统一目标 | 不逐条过 bug，回答为什么集中出现、哪些会反复出现、下一版怎么减少 | 对齐复盘目标 |
| 10 分钟 | 三期趋势复盘 | 产品讲版本目标变化，开发补充实现复杂度变化 | 明确问题从 UI 到交易闭环再到状态机 |
| 20 分钟 | 根因分类 | 需求口径、设计交付、开发自测、验收机制四类 | 归纳 Top 3 根因 |
| 15 分钟 | 下期机制调整 | 定活动、订单、履约三张回归矩阵 | 明确下期必测清单 |
| 10 分钟 | 行动项确认 | 产品、开发、测试/产品各自责任 | 形成会后待办 |

## 3. 三期趋势复盘

### v1.1：基础体验和 UI 一致性

观察到的问题类型：
- 输入框、价格、小数点、退款说明、页面滑动等基础交互。
- 团长管理页按钮、头像、券码核销、图标、顶部样式等视觉一致性。
- 发布活动标题、描述、图片数量、报名截止时间等基础规则。
- 未登录头像昵称、渠道访问、订单样式、核验结果等前台展示。

复盘判断：`v1.1` 主要暴露“页面级完成度”和“基础规则是否按设计稿落地”。这类问题适合通过设计走查、开发自测清单和提测前 UI 对照来减少。

### v1.2：交易链路和业务闭环

观察到的问题类型：
- 发布团购样式、售卖时间判断、规格名称和规格组限制。
- 限购步进器、限购处理、报名信息为空、订单详情默认商品名。
- 随机退款文案、退款按钮、团长订单管理中的退款/物流展示。
- 物流订单缺少发货时间、收货时间、订单字段等。

复盘判断：`v1.2` 已经进入“交易闭环是否跑通”的阶段。问题不再只是页面像不像，而是发布、下单、退款、物流、订单详情之间的规则是否一致。

### v1.3：复杂状态和边界组合

观察到的问题类型：
- 自提、卡券、活动参与完成后的状态展示。
- 自提售卖时间、售后申请、退款金额、订单待支付/待核销状态。
- 售卖结束商品编辑、活动重新发布、活动开始时间不可选过去时间。
- 团长底栏订单气泡、售后按钮、退款/未退款文案、限购库存校验。

复盘判断：`v1.3` 最值得关注的是“状态组合带来的系统性风险”。交易、履约、售后类问题优先级高于纯视觉问题，因为它们直接影响用户信任和真实订单。

## 4. 根因分类

建议现场按四类归因，不按个人归因。

| 根因 | 典型表现 | 下一步改进 |
| --- | --- | --- |
| 需求口径不清 | 状态、边界、文案、异常场景没有明确规则 | 产品在需求中补状态规则、异常分支和验收口径 |
| 设计交付不完整 | 缺少空态、错误态、售后态、已结束态、重新发布态 | 设计/产品交付时补关键状态截图或文字规则 |
| 开发自测不足 | 主流程可用，但组合状态、历史数据、边界值漏测 | 开发提测前按矩阵自测并记录结果 |
| 验收机制不足 | 测试清单没有沉淀成版本回归矩阵 | 产品/测试共同维护固定回归表，每期复用 |

建议会中锁定 Top 3 根因：
- 状态规则没有在需求阶段被完整前置。
- 开发自测更偏主流程，缺少组合状态覆盖。
- 验收清单没有沉淀为可复用的回归矩阵。

## 5. 下期必测回归矩阵

### 活动状态矩阵

| 状态 | 必测点 |
| --- | --- |
| 未开始 | 是否允许编辑、是否允许购买/报名、前台提示是否正确 |
| 进行中 | 下单、报名、分享、库存/名额变化是否正常 |
| 已结束 | 是否禁止继续购买/报名，已结束文案和按钮状态是否一致 |
| 重新发布 | 时间、库存、订单入口、历史状态是否被正确处理 |
| 编辑中 | 已有订单/报名情况下，哪些字段可改、哪些字段不可改 |

### 订单状态矩阵

| 状态 | 必测点 |
| --- | --- |
| 待支付 | 前台展示、倒计时、取消/继续支付入口 |
| 已支付 | 订单详情、履约入口、售后入口 |
| 待核销 | 券码、自提码、核销状态和团长侧展示 |
| 已完成 | 完成状态、复购/再次购买入口、售后限制 |
| 退款中 | 退款金额、退款说明、订单状态同步 |
| 已退款 | 已退款文案、按钮隐藏、收益/结算影响 |
| 售后中 | 售后申请、售后进度、平台/团长处理状态 |

### 履约方式矩阵

| 履约方式 | 必测点 |
| --- | --- |
| 自提 | 自提时间、提货状态、售后申请、过期/未提处理 |
| 物流 | 发货时间、收货时间、物流字段、退款/售后状态 |
| 卡券 | 券码生成、核销、过期、已使用/未使用状态 |
| 履约 + 售后组合 | 自提/物流/卡券分别进入退款、售后、完成后的状态一致性 |

## 6. 会后行动项

| 角色 | 行动项 | 验收方式 |
| --- | --- | --- |
| 产品 | 补齐活动、订单、履约、售后状态规则和验收口径 | 每个需求必须带主流程、异常状态、不可操作状态 |
| 开发 | 提测前按回归矩阵完成自测 | 提测说明中附自测覆盖范围和已知风险 |
| 测试/产品 | 共同维护三张固定回归矩阵 | 每期测试不从零写清单，优先复用矩阵并增补新场景 |
| 产品/开发 | 对 `v1.3` 高风险项做二次确认 | 退款金额、售后申请、限购库存、订单状态、自提时间、活动结束/重发状态全部过一遍 |

## 7. 现场可用话术

- 问题数量变多，本质是业务闭环变复杂了。
- `v1.3` 最值得复盘的不是单点 bug，而是状态组合带来的系统性风险。
- 下一版不能只靠测试同学发现问题，要把产品规则、开发自测、回归验收前置。
- 交易、售后、履约类问题优先级高于纯视觉问题，因为它们直接影响用户信任和真实订单。

## 8. 会后沉淀模板

会后建议沉淀一页结论：

- 三期 bug 趋势：`v1.1` 基础体验，`v1.2` 交易链路，`v1.3` 复杂状态。
- Top 3 根因：状态规则前置不足、组合状态自测不足、回归矩阵未沉淀。
- 下期必测矩阵：活动状态、订单状态、履约方式。
- 角色动作：产品补规则，开发补自测，测试/产品维护矩阵。
- 待确认风险：退款金额、售后申请、限购库存、订单状态、自提时间、活动结束/重发状态。

## 9. 备注

- 飞书表格主体为虚拟渲染，当前总结基于 `v1.1`、`v1.2`、`v1.3` 视图的可见记录和滚动抽样；会议现场可用飞书原表补充具体案例。
- 本材料只用于复盘与协作机制改进，不作为新功能 PRD，不扩大当前产品范围。
