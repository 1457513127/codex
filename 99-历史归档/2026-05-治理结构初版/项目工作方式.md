# 项目工作方式

本文定义和趣项目长期策划、讨论、交付和记录方式。它是新会话的第一阅读入口。

## 1. 基本原则

- 唯一项目事实源是本仓库文件，不是某一次聊天记录。
- Codex 对话可以按专题拆分，但所有稳定结论必须回写到仓库。
- Notion 用于外部协作记录和更新同步，GitHub 仓库用于保存可追溯项目资产。
- PPT、原型、产品蓝图都是交付物，不是随手草稿；只有形成明确口径后再修改。

## 2. 推荐的 Codex 使用方式

不建议在 `New project` 中只用一个超长对话承载所有复杂讨论。建议采用：

`一个项目仓库 + 多个专题对话 + 文件化长期记忆`

推荐按专题创建对话：

- 产品定位与商业模式
- 用户分层与新手冷启动
- 活动变现与货盘
- 商品库与开团发布
- 分销帮卖与收益规则
- 订单售后与结算
- 多端承载与 App 路线
- PPT 与会议材料
- 原型与界面走查
- 市场研究与竞品分析

每个专题对话结束前，应沉淀到以下文件之一：

- 稳定结论写入 `docs/PROJECT_MEMORY.md`。
- 重大决策写入 `00-项目治理/决策记录.md`。
- 未解决问题写入 `00-项目治理/待确认问题.md`。
- 专题材料写入对应规划文档或新增专题文档。
- 交付改动写入 `docs/UPDATE_LOG.md` 并同步 Notion。

## 3. 每轮工作分类

开始前先判断本轮属于哪一类：

| 类型 | 产出 | 是否默认改 PPT/原型 |
| --- | --- | --- |
| 战略定位 | 方向判断、边界、决策记录 | 否 |
| 功能规划 | 蓝图、模块说明、优先级 | 否 |
| PRD 拆解 | 页面、对象、流程、验收标准 | 否 |
| 原型/PPT | 可点击原型、演示稿、走查脚本 | 是，仅限用户明确要求 |
| 市场研究 | 调研结论、证据边界、引用来源 | 否 |
| 执行任务 | 文件修改、验证、提交、推送 | 按任务范围 |

仅讨论和规划不自动修改 PPT/原型。只有形成明确版本口径，且用户要求更新交付物时，才进入 PPT 或原型修改。

## 4. 交付流程

一次有效交付必须包含：

1. 读取治理入口、上下文总览和项目记忆。
2. 明确本轮目标和不做范围。
3. 小范围修改必要文件。
4. 执行适合本轮的验证。
5. 更新 `docs/UPDATE_LOG.md`。
6. 同步 Notion。
7. 创建聚焦 GitHub 提交并推送。

## 5. 长期维护习惯

- 不把所有历史都塞进 README。
- 不把临时讨论写入长期记忆。
- 不把尚未确认的设想写成已决策。
- 不让 PPT 和原型超前于产品蓝图。
- 不为每个平台、每个话题复制一份项目目录。
