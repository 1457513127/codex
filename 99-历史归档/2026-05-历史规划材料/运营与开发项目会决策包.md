## 1. 本次会议目标

这不是一次功能收集会，而是一次增长验证启动会。会议结束前需要回答：

> 未来 3 个月，公司是否愿意用可控资源验证“团长轻主页 + 一级帮卖分销”能够带来增量订单，并且退款、佣金和履约责任可管控？

## 3. 会上先对齐的事实

| 事实 | 会议意义 |
| --- | --- |
| 已有 227 场活动、6,207 笔订单 | 不再讨论“是否存在任何活动需求”，而讨论如何验证增长 |
| 退款订单比例约 28.3% | 不允许只讨论拉单和佣金而忽略退款质量 |
| 当前团长帮卖没有系统归因数据 | 不能用感受决定扩量，先补证据 |
| 当前平台中家长不知道团长身份且活动分散在链接中 | 可作为主页假设来源，但需数据验证 |
| 商品和泛服务会新增供应链/售后复杂度 | 不与亲子分销一期同时开发 |

## 4. 需要现场拍板的范围

### 4.1 提交会议确认的 MVP

| 业务路径 | MVP 能力 | 是否同意进入开发评估 |
| --- | --- | --- |
| 家长 | 团长主页查看活动、活动详情显示主办方/团长关系、按原流程报名支付与售后 | 待确认 |
| 团长 | 审核入驻、个人主页、选择可推广活动、专属分享、推广订单、佣金状态 | 待确认 |
| 运营 | 主办方/活动审核、是否允许分销、暂停分销、团长审核、渠道和退款数据导出 | 待确认 |
| 支付/结算 | 佣金延迟生效、结算前退款取消、结算后退款方案确认 | 待确认 |

### 4.2 明确排除

- 商品发布、商品选品、库存、物流和退换货。
- 多级分销、邀请团长获得奖励、公开佣金排行。
- 团长内容社区、帖子、评论和成员互动。
- 未完成支付方案确认时上线实际佣金自动结算。

## 5. 推荐会议议程

| 时间 | 议题 | 主持/输入 | 会议输出 |
| ---: | --- | --- | --- |
| 0-10 分钟 | 为什么从交易验证进入增长验证 | 产品：总览与数据基线 | 统一问题定义 |
| 10-25 分钟 | 运营现状核实 | 运营：现有帮卖流程、团长数量、售后痛点 | 现行流程事实清单 |
| 25-40 分钟 | 可点击界面走查 | 产品：家长、团长、运营路径 | 功能范围异议清单 |
| 40-55 分钟 | 技术与数据可行性 | 开发：角色、归因、佣金、历史订单、埋点 | 技术依赖与粗估 |
| 55-65 分钟 | 支付/退款/结算边界 | 支付/财务/法务 | 结算确认任务与上线前置 |
| 65-80 分钟 | 3 个月试点范围与指标 | 产品 + 运营 | 活动、团长、指标、停止条件 |
| 80-90 分钟 | 拍板和责任人 | 全员 | Go/No-Go、负责人、日期 |

## 6. 运营部门必须回答的问题

| 问题 | 为什么必须问 | 会后产物 |
| --- | --- | --- |
| 目前有多少帮卖团长，活跃程度如何？ | 判断 10 名试点是否现实 | 首批团长名单 |
| 帮卖订单目前如何识别、佣金如何手工结算？ | 找出归因和对账的真实流程 | 当前流程图与风险点 |
| 哪些活动最适合群内推广，哪些退款/取消最多？ | 确定白名单和黑名单 | 首轮 20 场候选活动 |
| 家长最常问的售前/售后问题是什么？ | 决定主页和详情页信息优先级 | FAQ 与展示字段 |
| 外部主办方资料目前有什么、缺什么？ | 判断代售准入落地成本 | 准入资料模板 |

## 7. 开发部门必须回答的问题

| 问题 | 判断内容 |
| --- | --- |
| 现有账号/权限模型能否拆分主办方与团长身份？ | 是否需要角色升级和迁移 |
| 现有订单是否可增加 `leader_id`、`organizer_id` 与入口来源？ | 归因技术成本 |
| 分享卡片/小程序路径参数如何携带和锁定归因？ | 归因冲突规则和埋点 |
| 退款、核销、完成状态是否能驱动佣金状态？ | 佣金账本实现方式 |
| 儿童和保险敏感字段如何限制团长可见范围？ | 数据权限与安全 |
| 现有页面可复用多少？ | 研发粗估与演示范围 |
| 数据导出和周报所需字段何时可具备？ | 试点可衡量性 |

## 8. 支付与合规必须确认的问题

| 问题 | 不能跳过的原因 |
| --- | --- |
| 团长以个人还是主体身份接收佣金？ | 微信支付分账回退对接收方类型有边界 |
| 佣金是否一律延迟到履约完成并过退款窗口？ | 降低退款后的资金追回风险 |
| 已结算后退款如何回退或冲抵？ | 没有路径不得上线真实结算 |
| 团长主页及分享如何披露推荐关系？ | 带购买入口的推广内容需满足披露要求 |
| 团长能看哪些订单数据？ | 儿童和保险信息须最小必要处理 |

## 9. 建议当场做出的八项决策

| 编号 | 决策 | 推荐选项 |
| ---: | --- | --- |
| D1 | 是否进入增长试点 | `Go：仅亲子活动受控一级帮卖` |
| D2 | 角色模型 | `公司/平台 + 主办方 + 团长 + 家长` |
| D3 | 团长主页范围 | `身份 + 关系披露 + 活动货架 + 分享归因` |
| D4 | 分销层级 | `仅一级佣金` |
| D5 | 佣金生效节点 | `履约完成且过退款窗口` |
| D6 | 首批供给 | `公司自营 + 精选外部亲子活动，不含高风险扩量` |
| D7 | 试点规模 | `10 名团长、20 场活动` |
| D8 | 商品/泛服务 | `本阶段不开发，另案评估` |

## 10. 项目会展示路径

会议中只展示能影响决策的页面：

1. 家长从微信群进入团长主页，看到团长推荐关系与可报名活动。
2. 家长进入活动详情，看到实际主办方、安全、保险与退款规则。
3. 团长进入推广工作台，选择可推广活动并查看佣金状态。
4. 运营在审核台限制高退款活动、审核团长并查看归因漏斗。
5. 用退款场景演示为何佣金不能支付后立即可提现。

界面说明见：[演示界面走查脚本](../03-演示界面/演示界面走查脚本.md)。

## 11. 会议结束标准

会议不能以“大家先看看”结束。至少形成：

- `Go / No-Go / 补证据后再定` 的结论。
- 角色、MVP 范围、明确不做事项。
- 数据口径、支付结算、技术评估、首批运营试点四类责任人与完成日期。
- 下一次评审时间及所需演示/数据输出。
